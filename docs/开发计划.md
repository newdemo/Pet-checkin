## 1. 阶段目标

在微信开发者工具中可运行，跑通完整业务闭环：

```
家长设置任务 → 孩子打卡 → 家长确认 → 获得道具 → 喂养宠物 → 成长升级 → 记录更新
```

**本阶段不做：** 支付、社交、排行榜、复杂动画、多账号体系。

## 5. 技术要点

### 5.1 Mock 优先策略（实际实现）

```
开发顺序: UI → Mock 闭环 → 云开发替换
            ✅      ✅           ❌
```

- 所有业务逻辑经由 `services/storage.js` 调用
- Mock 层使用 `uni.setStorageSync` + 内存缓存实现持久化
- 通过 `data.taskDate` 判断跨日，自动重置每日任务

### 5.2 状态同步（实际实现）

- 各页面在 `onShow` 时调用 services 重新读取最新数据
- 操作成功后写 storage，下一次 onShow 自动刷新
- ⚠️ 无 Pinia Store，页面间无实时响应式同步

### 5.3 升级逻辑

参见 [constants/pet.js](file:///Users/zs/Desktop/Trae_实验室/宠物打卡/frontend/src/constants/pet.js) 和 [services/pet.js](file:///Users/zs/Desktop/Trae_实验室/宠物打卡/frontend/src/services/pet.js)

### 5.4 每日任务生成

采用懒加载生成：打开任意页面时 `getAppData()` 自动检测跨日并重新生成当日任务。

## 6. 风险与应对

| 风险 | 影响 | 当前状态 |
|------|------|----------|
| 云开发环境未开通 | 无法联调云端 | 不影响，Mock 模式已可演示完整闭环 |
| 儿童操作误触家长功能 | 任务被误改 | ⚠️ 家长入口无验证，需后续增加确认弹窗 |
| 同日任务重复生成 | 数据重复 | ✅ 已通过 templateId + taskDate 唯一约束 |
| 确认发奖重复执行 | 道具多发 | ⚠️ 当前仅在 reviewCheckin 内校验 status，无独立幂等标记 |
| 插画资源不足 | 界面简陋 | 当前使用 emoji + 纯色块占位 |
| 目录结构重复 | 同事修改错误文件 | ⚠️ frontend/ 根目录与 src/ 存在旧版重复文件 |
| 无全局状态管理 | 多页面数据不同步 | 当前通过 onShow 重新读取规避，后续需引入 Pinia |
