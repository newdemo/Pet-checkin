## 1. 本阶段目标

在微信开发者工具中可运行，跑通完整业务闭环：

```
家长设置任务 → 孩子打卡 → 家长确认 → 获得道具 → 喂养宠物 → 成长升级 → 记录更新
```

**技术约束：** UniApp + Vue3 + 微信小程序；本地 Mock（无云开发）；不涉及支付、社交、排行榜、多账号体系。

**接手基线：** 项目从 Cursor 环境迁移至 Trae IDE，`frontend/src/` 为唯一主线源码目录。Mock 模式下可编译运行，完整业务闭环可用，但存在以下遗留问题：

- 打卡直接完成并发放奖励，缺少家长审核环节
- 任务管理仅支持开关，不支持新增/编辑
- `frontend/` 根目录与 `frontend/src/` 存在旧版重复文件
- 未引入 Pinia 全局状态管理

## 4. 发现的规则问题

### 4.1 状态标记格式不统一

**问题：** 项目初期使用符号标记（⬜/🔄/⚠️/✅），规则更新后要求使用文字标记（待开发/开发中/待验收/已完成）。

**影响：** 状态标记需要全量迁移，涉及 TODO.md 中 23 个任务 + 图例表。

**解决：** 在 P1-5 收尾阶段一次性完成迁移。

**教训：** 项目启动时应先确定状态标记规范，避免后期迁移成本。

### 4.2 完成度计算公式变更

**问题：** 旧公式（98% = 23.5/24 任务）与新规则公式（30.4% = 700% ÷ 23 任务）差异巨大。

**原因：** 旧公式按"主观完成度"估算，新规则严格按状态权重计算（已完成=100%, 待验收=80%, 开发中=50%, 待开发=0%）。

**影响：** 完成度从 98% 骤降至 30.4%，虽然 Phase 1 (P0+P1) 实际已完成 100%，但项目总完成度受 P2/P3/P4 大量待开发任务拉低。

**教训：** 完成度公式应在项目初期确定，中途变更会造成数据跳跃。新公式更客观，但需要向用户解释清楚计算逻辑。

### 4.3 规则文件在开发过程中更新

**问题：** `Rules/项目文档维护规则.md` 在 P1-5 开发过程中被用户更新，导致需要重新读取规则并检查当前流程是否受影响。

**影响：** 规则变更后需要暂停当前流程，重新判断任务闭环状态，增加了流程复杂度。

**教训：** 规则变更应尽量在任务间隙进行，避免打断正在执行的任务流程。

## 5. 发现的流程问题

### 5.1 P1-4 开发中断恢复

**问题：** 在 `parent/tasks/index.vue` 中查找 `.btn-settings` 时疑似卡住，导致开发中断。

**恢复流程：**
1. 执行 `git status` 检查当前状态
2. 列出已修改文件
3. 检查已完成改动
4. 检查是否存在语法错误或半成品代码
5. 不 commit、不 push
6. 输出中断点和恢复建议
7. 等待用户确认后继续

**结果：** 确认无语法错误、无半成品代码，仅缺 CSS 样式，安全恢复后继续开发。

**教训：** 开发中断时应立即执行安全恢复流程，不要盲目继续修改代码。安全恢复流程已形成可复用的 SOP。

### 5.2 P1-5 删除后编译失败

**问题：** 删除 20 个旧文件后 `uni build` 报错 `Could not resolve "../../constants/pet" from "src/components/TaskFormModal.vue"`。

**原因：** `TaskFormModal.vue` 的导入路径 `../../constants/pet` 在删除旧 `constants/` 目录后解析失败。实际应指向 `src/constants/pet`（路径为 `../constants/pet`）。

**解决：** 修改导入路径为 `../constants/pet`，编译通过。

**教训：** 删除操作前应检查所有导入引用，确保删除的目录中没有被其他文件引用的模块。高风险删除操作后必须立即执行编译验证。

### 5.3 P1-5 收尾流程过长

**问题：** P1-5 从分析到最终收尾经历了多轮迭代：
1. 分析报告（可删除/疑似/禁止删除 + 风险分析）
2. 用户确认
3. 执行删除
4. 编译验证（发现导入路径错误）
5. 修复导入路径
6. 文档更新（TODO/CHANGELOG/开发计划）
7. 状态标记迁移
8. 完成度重算
9. Git Commit
10. 用户确认 Push

**教训：** 高风险任务（删除）的收尾流程天然比普通任务长。应在任务规划时预估收尾步骤数量，避免用户预期偏差。

### 5.4 DeleteFile 工具失败

**问题：** 尝试用 DeleteFile 工具批量删除 20 个文件时，部分文件无法移动到废纸篓。

**解决：** 改用 `rm -rf` 命令成功删除。

**教训：** DeleteFile 工具对大文件或特殊文件可能失败，`rm -rf` 是更可靠的备选方案。但 `rm -rf` 属于高风险命令，需要用户确认后执行。

## 7. 本次项目的经验总结

### 7.1 技术经验

1. **easycom 优于手动 import：** UniApp 的 easycom 自动注册机制可以避免组件重复注册导致的编译冲突（P0-1 修复经验）。

2. **Pinia 改造应尽早进行：** P1-1 引入 Pinia 后，后续 P1-2/P1-3/P1-4 的开发效率明显提升，store 驱动的页面间数据同步比手动 `onShow` 读取更可靠。

3. **Mock 优先策略有效：** 在无云开发环境的情况下，`uni.setStorageSync` + 内存缓存的 Mock 层完整支撑了 Phase 1 全部功能开发和验收。

4. **编译验证是安全网：** P1-5 删除后编译失败被立即发现并修复，证明每次修改后执行编译验证的必要性。

### 7.2 流程经验

1. **安全恢复流程价值高：** P1-4 开发中断时执行的安全恢复流程避免了盲目修改代码，该流程可复用于未来的开发中断场景。

2. **高风险任务需要分阶段执行：** P1-5 的删除操作分为分析→确认→执行→验证四个阶段，每个阶段都有明确的用户确认点，避免了误删风险。

3. **文档同步不能省略：** 每次任务完成后同步更新 TODO/CHANGELOG/开发计划，确保了项目状态在不同模型之间可无缝接续。

4. **Git 工作流规范：** 开发→编译→待验收→用户验收→文档更新→Git Commit 的流程保证了每次提交都是经过验证的完整版本。

### 7.3 协作经验

1. **用户验收是质量关卡：** BUG-01 和 BUG-02 由人工运行验证发现，说明 AI 开发后的功能需要人工验收来捕获边缘 case。

2. **规则驱动开发可行：** 项目规则文件（Rules/项目文档维护规则.md）作为唯一开发规范，在多次任务中有效约束了 AI 行为，避免了自行推断和跳过流程。

3. **断点机制保障接续：** TODO.md 中的「当前开发断点」字段确保了不同模型或不同会话之间可以准确接续开发。

## 8. 后续项目可复用经验

### 8.1 可复用的流程模板

**标准开发流程（已验证）：**
```
需求确认 → 开发 → 编译验证 → 待验收 → 用户验收 → 文档更新 → 完成度更新 → Git Commit → 输出收尾报告 → （可选）高风险操作 → （可选）Git Push → 进入下一任务
```

**安全恢复流程（已验证）：**
```
git status → 列出已修改文件 → 检查已完成改动 → 检查语法错误/半成品代码 → 不 commit/push → 输出中断点和恢复建议 → 等待用户确认
```

**高风险删除流程（已验证）：**
```
功能开发完成 → 编译验证通过 → 用户验收通过 → 文档更新完成 → Git Commit 完成 → 删除分析报告 → 用户确认 → 执行删除 → 编译验证 → 文档更新 → Git Commit
```

### 8.2 可复用的文档模板

- `docs/TODO.md` — 项目驾驶舱模板（版本/阶段/完成度/断点/已知问题）
- `docs/CHANGELOG.md` — 语义化版本变更日志模板
- `docs/开发计划.md` — 分 Sprint 任务拆解模板
- `Rules/项目文档维护规则.md` — 项目规则模板

### 8.3 可复用的技术模式

- **easycom 组件注册模式：** 在 `pages.json` 中配置 `easycom` 自动注册，避免手动 import
- **Pinia store 模式：** store 封装业务逻辑，页面只负责 UI 渲染和事件分发
- **Mock 数据层模式：** `services/storage.js` 作为统一数据访问层，通过 `uni.setStorageSync` + 内存缓存实现持久化
- **角色切换模式：** role store + 持久化 + 确认弹窗 + 返回按钮

## 9. P2 阶段优化建议

### 9.1 优先级排序

| 优先级 | 任务 | 理由 |
|--------|------|------|
| **P2-1** | Sass @import 迁移 | 消除 6 条编译警告，改善开发体验，改动小风险低 |
| **P2-2** | 道具不足引导 | 提升儿童用户体验，减少困惑 |
| **P2-3** | 首页加载优化 | 消除双重触发，提升首屏性能 |
| **P2-4** | 死代码清理 | 减少代码维护负担 |
| **P2-5** | 颜色变量统一 | 提升代码可维护性，为后续主题切换打基础 |

### 9.2 建议执行顺序

```
P2-1（Sass 迁移，低风险快速见效）
  ↓
P2-2（道具不足引导，UX 提升）
  ↓
P2-3（首页加载优化，性能提升）
  ↓
P2-4（死代码清理，代码健康）
  ↓
P2-5（颜色变量统一，可维护性）
```

### 9.3 P2 阶段注意事项

1. **P2 任务均为体验优化，不涉及新功能开发**，每个任务改动范围小，可快速迭代。
2. **P2-1 建议优先执行**，因为 Sass 废弃警告在每次编译时都会出现，消除后开发体验更好。
3. **P2-5 颜色变量统一**涉及多个页面，改动范围较大，建议放在 P2 最后执行。
4. **BUG-01/BUG-02 暂不修复**，作为 P2 测试验证阶段的 AI 自动测试可行性验证样本。

### 9.4 P2 完成后的里程碑

P2 全部完成后，项目将进入 P3（Cloud & Launch）阶段，涉及云开发接入、真机测试、微信审核等上线准备工作。
