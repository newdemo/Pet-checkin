> 历史参考文档：本文包含早期数据库方案、旧集合命名、snake_case 字段和旧状态流转。P4 阶段执行依据以 [P4-2A_最小数据库集合设计.md](file:///Users/zs/Desktop/Trae_实验室/宠物打卡/docs/P4-2A_最小数据库集合设计.md) 为准。P4-2B 不得依据本文创建 `family`、`streak_stat`、`daily_summary`、`inventory_log`、`task_template`、`daily_task`、`checkin` 等历史集合。

## 1. 设计原则

- **按家庭隔离数据**：MVP 一个微信账号对应一个家庭单元，以 `openid` 作为数据归属键。
- **软删除**：任务等配置类数据使用 `enabled` 字段开关，不做物理删除。
- **日维度任务实例**：每日根据任务模板生成当日任务记录，便于打卡状态追踪。
- **幂等发奖**：家长确认打卡时通过 `checkin_id` 保证道具只发放一次。

---

## 2. ER 关系概览

```
family (家庭/用户)
  ├── 1:1 ── pet (宠物)
  ├── 1:N ── task_template (任务模板)
  ├── 1:N ── daily_task (每日任务实例)
  │              └── 1:1 ── checkin (打卡记录)
  └── 1:N ── inventory_log (道具流水，可选)

streak_stat (连续打卡统计) ── 归属 family
```

## 3. 集合（表）设计

### 3.1 family — 家庭用户

存储用户基础信息与角色偏好。

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `_id` | string | 自动 | 文档 ID |
| `openid` | string | 是 | 微信用户唯一标识，索引 |
| `role` | string | 是 | 当前模式：`child` / `parent`，默认 `child` |
| `created_at` | date | 是 | 创建时间 |
| `updated_at` | date | 是 | 更新时间 |

**索引：**

- `openid`：唯一索引

---

### 3.2 pet — 宠物

每个家庭一条宠物记录。

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `_id` | string | 自动 | 文档 ID |
| `family_id` | string | 是 | 关联 family._id |
| `name` | string | 是 | 宠物名称，默认「小萌」 |
| `level` | number | 是 | 等级 1~5，默认 1 |
| `growth_value` | number | 是 | 当前等级内成长值，默认 0 |
| `hunger` | number | 是 | 饥饿值 0~100，默认 80 |
| `cleanliness` | number | 是 | 清洁值 0~100，默认 80 |
| `mood` | number | 是 | 心情值 0~100，默认 80 |
| `avatar_stage` | number | 是 | 形象阶段，同 level，默认 1 |
| `created_at` | date | 是 | 创建时间 |
| `updated_at` | date | 是 | 更新时间 |

**索引：**

- `family_id`：唯一索引（一个家庭一只宠物）

**成长值升级阈值（应用层常量）：**

```javascript
const LEVEL_THRESHOLDS = {
  1: 50,   // Lv.1 → Lv.2 需要 50
  2: 70,   // Lv.2 → Lv.3 需要 70（累计 120）
  3: 100,  // Lv.3 → Lv.4 需要 100（累计 220）
  4: 130,  // Lv.4 → Lv.5 需要 130（累计 350）
};
```

### 3.5 daily_task — 每日任务实例

每天根据启用的 `task_template` 生成，记录当日打卡状态。

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `_id` | string | 自动 | 文档 ID |
| `family_id` | string | 是 | 关联 family._id |
| `template_id` | string | 是 | 关联 task_template._id |
| `task_date` | string | 是 | 日期 `YYYY-MM-DD` |
| `name` | string | 是 | 冗余任务名（快照） |
| `icon` | string | 是 | 冗余图标 |
| `reward_type` | string | 是 | 冗余奖励类型 |
| `reward_amount` | number | 是 | 冗余奖励数量 |
| `status` | string | 是 | `pending` / `waiting` / `done` |
| `checkin_id` | string | 否 | 关联 checkin._id |
| `created_at` | date | 是 | 创建时间 |
| `updated_at` | date | 是 | 更新时间 |

**状态说明：**

| status | 含义 |
|--------|------|
| `pending` | 未完成 |
| `waiting` | 已打卡，待家长确认 |
| `done` | 家长已确认，已完成 |

**索引：**

- `family_id + task_date`：组合索引
- `family_id + task_date + template_id`：唯一索引（同日同模板仅一条）

**生成策略：**

- 用户当日首次打开「今日任务」或定时云函数在 0 点生成。
- 仅同步 `enabled=true` 的模板。
- 已存在的 `daily_task` 不重复生成。

### 3.7 streak_stat — 连续打卡统计

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `_id` | string | 自动 | 文档 ID |
| `family_id` | string | 是 | 关联 family._id |
| `current_streak` | number | 是 | 当前连续天数，默认 0 |
| `longest_streak` | number | 是 | 历史最长连续天数，默认 0 |
| `total_completed` | number | 是 | 累计完成任务数，默认 0 |
| `last_checkin_date` | string | 否 | 最近一次有效打卡日期 |
| `updated_at` | date | 是 | 更新时间 |

**索引：**

- `family_id`：唯一索引

**连续天数计算规则：**

- 家长确认当日至少 1 个任务 → 该日记为有效打卡日。
- 若 `last_checkin_date` 为昨天 → `current_streak + 1`。
- 若间隔 > 1 天 → `current_streak` 重置为 1。
- 同日多次确认不重复累加连续天数。

## 4. 核心业务流程与数据变更

### 4.1 孩子打卡

```
输入: daily_task_id
前置: daily_task.status === 'pending'

变更:
  1. 创建 checkin 记录 (status: waiting)
  2. 更新 daily_task.status = 'waiting', checkin_id = 新记录ID
```

### 4.2 家长确认

```
输入: checkin_id
前置: checkin.status === 'waiting' && !checkin.reward_granted

变更（事务或云函数原子操作）:
  1. checkin.status = 'approved', reviewed_at = now, reward_granted = true
  2. daily_task.status = 'done'
  3. inventory[reward_type] += reward_amount
  4. 更新 streak_stat（连续天数、累计完成数）
  5. 可选：更新 daily_summary
```

### 4.3 家长驳回

```
输入: checkin_id

变更:
  1. checkin.status = 'rejected'
  2. daily_task.status = 'pending', checkin_id = null
```

### 4.4 使用道具养成

```
输入: action (feed/wash/play), 对应道具类型
前置: inventory[item] >= 1

变更:
  1. inventory[item] -= 1
  2. pet 对应状态值 += 20（上限 100）
  3. pet.growth_value += 5
  4. 若 growth_value >= 当前等级阈值 → level += 1, growth_value -= 阈值, avatar_stage 更新
```

## 5. 云开发权限规则（草案）

```json
{
  "family": {
    "read": "doc.openid == auth.openid",
    "write": "doc.openid == auth.openid"
  },
  "pet": {
    "read": "doc.family_id in getFamilyIds()",
    "write": "doc.family_id in getFamilyIds()"
  }
}
```

> MVP 阶段可将读写逻辑集中在云函数内，数据库规则设为仅云函数可写，降低前端直接操作风险。
