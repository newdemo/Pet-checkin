# **项目文档维护规则**

## **规则说明**

本规则为项目唯一开发规范。

Trae、Cursor、Codex、CodeBuddy 及其他 AI 模型执行任务时必须严格遵守。

规则优先级：

用户明确指令
 ＞
 项目规则
 ＞
 默认开发习惯

禁止：

- 自行推断开发进度
- 跳过规则执行
- 未经确认执行高风险操作

------

## **文档同步规则**

每次开发任务完成后必须检查：

### **产品文档**

检查是否影响：

- 产品需求
- 功能说明
- 业务流程

如有变化必须同步更新。

### **技术文档**

检查是否影响：

- 页面结构
- 路由结构
- 数据库设计
- 接口设计
- 项目目录结构

如有变化必须同步更新。

### **项目管理文档**

自动维护：

- docs/TODO.md
- docs/开发计划.md
- docs/CHANGELOG.md

------

## **任务管理规则**

### **优先级**

统一使用：

- P0（MVP核心功能）
- P1（重要功能）
- P2（体验优化）
- P3（后续规划）

### **状态**

统一使用：

- 待开发
- 开发中
- 待验收
- 已完成

禁止使用其它状态。

### **Sprint**

所有任务必须归属于 Sprint。

不得存在未归属 Sprint 的任务。

------

## **验收管理规则**

### **开发完成不等于任务完成**

功能开发结束后：

状态必须标记为：

⚠️ 待验收

### **验收通过条件**

仅当用户明确反馈：

- 验收通过
- 测试通过
- 功能符合预期

后，方可修改为：

✅ 已完成

### **未验收任务限制**

未验收通过的任务：

- 不计入已完成任务
- 不允许标记里程碑
- 不允许创建正式 Commit
- 不允许从 TODO 中移除

------

## **完成度计算规则**

### **项目总完成度**

项目完成度必须自动计算。

状态权重：

- 已完成 = 100%
- 待验收 = 80%
- 开发中 = 50%
- 待开发 = 0%

计算公式：

项目总完成度

=

所有任务状态权重总和

÷

总任务数

### **完成度输出要求**

必须输出：

- 项目总完成度
- 当前阶段完成度

### **禁止行为**

禁止根据：

- P0/P1权重
- MVP权重
- Sprint权重
- 业务价值
- 主观判断

进行人工加权。

允许单独展示：

- MVP完成度
- Sprint完成度
- P0完成度
- P1完成度

但不得替代项目总完成度。

------

## **CHANGELOG规则**

每次更新必须记录：

### **新增**

新增功能

### **修改**

修改功能

### **修复**

Bug修复

### **涉及文件**

本次修改文件列表

------

## **TODO规则**

TODO 必须长期保留：

- 已完成
- 开发中
- 待开发

已完成任务不得删除。

仅废弃任务允许归档。

------

## **开发断点规则**

每次任务结束后必须更新：

- 当前版本
- 当前阶段
- 最近完成
- 当前开发断点
- 下一步任务

写入：

docs/TODO.md

确保项目可在不同模型之间无缝接续开发。

------

## **Git规则**

### **Commit前置流程**

必须严格遵循：

开发完成

↓

编译通过

↓

待验收

↓

用户验收通过

↓

文档更新

↓

Git Commit

### **正式Commit条件**

必须满足：

- 用户验收通过
- TODO已更新
- CHANGELOG已更新
- 开发计划已更新

方可创建正式 Commit。

### **WIP Commit规则**

允许创建临时 Commit：

wip: xxx

仅用于过程保存。

WIP Commit 不视为正式版本。

### **Commit输出内容**

创建 Commit 后必须输出：

- Commit ID
- Commit Message
- 当前版本
- 当前开发断点

------

## **GitHub同步规则**

### **Push规则**

Git Push 属于发布行为。

未经用户明确确认：

禁止执行：

git push

### **Push授权关键词**

仅当用户明确表达：

- 推送
- 上传GitHub
- 同步仓库
- 发布版本

时允许执行 Push。

------

## **高风险任务规则**

以下任务属于高风险任务：

- 删除目录
- 删除页面
- 删除组件
- 删除服务层
- 删除数据库字段
- 删除配置文件
- 重构目录结构

### 删除分析复用规则

同一任务中：

如已生成删除分析报告并完成用户确认，

后续执行删除时不得重新全项目扫描。

应直接基于已确认的删除清单执行。

仅当：

- 文件发生新增
- 引用关系发生变化
- 用户明确要求重新扫描

时允许再次生成删除分析报告。

避免重复消耗上下文和模型额度。

### **删除前置条件**

必须先完成：

- 功能开发完成
- 编译验证通过
- 用户验收通过
- 文档更新完成
- Git Commit完成

之后才能进入删除分析。

### **删除分析报告**

必须输出：

- 可删除
- 疑似可删除
- 禁止删除
- 删除影响范围
- 回滚方案
- 预计删除数量

### **删除执行规则**

分析完成后：

必须等待用户确认。

未经确认不得执行删除。

### **禁止行为**

禁止：

- 验收前删除
- Commit前删除
- 未分析直接删除
- 未提供回滚方案直接删除

高风险任务执行顺序：

验收通过

↓

文档更新

↓

Git Commit

↓

删除分析

↓

用户确认

↓

执行删除

------

## **任务收尾规则**

任务完成后必须输出：

- 当前版本
- 当前开发阶段
- 项目总完成度
- 当前阶段完成度
- 本次完成内容
- 更新文档列表
- Commit信息
- 当前开发断点
- 下一步建议

------

## 项目状态读取规则

进行任务规划、闭环判断、完成度计算、下一步规划前。

必须优先读取：

- docs/TODO.md
- docs/开发计划.md
- docs/CHANGELOG.md

不得仅根据当前工作区文件状态判断项目进度。

不得忽略历史 Commit。

不得忽略已完成验收记录。

如当前执行结果与项目文档记录冲突：

以项目文档记录为准。

如项目文档与Git记录冲突：

必须明确指出冲突项并停止等待确认。


## **项目交接规则**

任何模型接手项目时必须优先读取：

- docs/TODO.md
- docs/开发计划.md
- docs/CHANGELOG.md

以 TODO.md 中的：

当前开发断点

作为唯一开发起点。

禁止自行推断开发进度。

------


## **自主推进规则**

每个任务结束后必须根据当前项目状态判断下一步。

优先检查：

- 是否需要验收
- 是否需要文档更新
- 是否需要完成度更新
- 是否需要 Git Commit
- 是否需要高风险分析
- 是否需要用户确认
- 是否允许进入下一任务

禁止停留在“任务完成”状态等待用户主动询问下一步。

必须主动输出：

- 当前阶段
- 已完成事项
- 未完成事项
- 下一步建议
- 是否需要用户确认

---

## 任务闭环规则

每次任务结束后必须判断：

当前任务是否真正闭环。

未闭环不得推荐进入下一任务。

必须明确输出：

- 当前阶段
- 已完成事项
- 未完成事项
- 当前阻塞项
- 下一步动作

如下一步存在唯一合理动作：

直接给出推荐执行指令。

如下一步需要用户确认：

明确停止并等待确认。

---

## 删除分析复用规则

同一任务中，如已生成删除分析报告并完成用户确认，后续执行删除时不得重新全项目扫描。

应直接基于已确认的删除清单执行。

仅当以下情况发生时允许重新扫描：

- 文件结构发生变化
- 引用关系发生变化
- 用户明确要求重新分析

禁止重复生成相同删除分析报告。

优先复用已有分析结果。

避免重复消耗模型额度和上下文资源。

---

## 规则变更生效规则

项目规则发生变更后，必须立即重新读取最新规则。

重新读取后必须检查：

- 当前任务是否受规则影响
- 当前流程是否需要调整
- 当前任务是否需要重新规划

规则变更仅影响后续步骤。

已完成步骤不得重复执行。

禁止因规则更新而重复扫描项目。

禁止因规则更新而重复执行已完成任务。

禁止因规则更新重新进行：

- 验收
- 文档更新
- 完成度计算
- Git Commit
- 删除分析

除非用户明确要求重新执行。
