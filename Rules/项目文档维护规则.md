【项目文档维护规则】

每次完成开发任务后：

【文档同步】

1. 检查是否影响产品需求文档
2. 检查是否影响页面结构
3. 检查是否影响数据库设计

如有变化则同步更新对应文档。

【项目管理】

自动维护：

- docs/开发计划.md
- docs/CHANGELOG.md
- docs/TODO.md

更新内容包括：

- 当前版本
- 当前开发阶段
- 当前完成度
- 已完成功能
- 开发中功能
- 待开发功能
- 风险项
- 下一步计划

【变更记录】

CHANGELOG 必须记录：

- 新增功能
- 修改功能
- 修复问题
- 涉及文件

【待办管理】

TODO 必须自动维护：

- P0（必须完成）
- P1（重要优化）
- P2（后续规划）

已完成项自动移出待办。

【任务收尾】

任务完成后必须输出：

- 当前版本
- 当前开发阶段
- 当前完成度
- 本次完成内容
- 更新文档列表
- 下一步建议

如任务属于里程碑节点，自动建议创建 Git Commit。

【迭代管理】

每个开发任务必须归属于一个迭代（Sprint）。

任务开始时：

- 标记所属 Sprint
- 标记任务优先级（P0/P1/P2/P3）
- 标记任务状态

任务状态统一为：

- 待开发
- 开发中
- 待验收
- 已完成

任务完成后：

1. 更新 docs/开发计划.md 中对应任务状态
2. 更新当前版本号
3. 更新项目完成度
4. 判断是否达到里程碑
5. 达到里程碑时建议创建 Git Commit
6. 自动推荐下一优先级最高的任务

不得存在未归属 Sprint 的开发任务。

---

【验收管理】

开发完成不等于任务完成。

所有功能开发结束后：

状态必须先标记为：

- 待验收

仅当用户明确反馈：

- 测试通过
- 验收通过
- 功能符合预期

后，方可修改状态为：

- 已完成

未验收通过的功能：

- 不计入项目完成度
- 不允许从 TODO 中移除
- 不允许标记为里程碑完成

验收通过后：

1. 更新 docs/开发计划.md
2. 更新 docs/CHANGELOG.md
3. 更新 docs/TODO.md
4. 更新项目完成度
5. 输出当前项目状态

输出格式：

- 当前版本
- 当前开发阶段
- 当前完成度
- 本次完成内容
- 更新文档列表
- 下一步建议

【完成度计算规则】

项目完成度不得主观估算。

必须根据：

已完成任务数
÷
（已完成任务数 + 开发中任务数 + 待开发任务数）

自动计算。

计算时：

- 已完成 = 100%
- 待验收 = 80%
- 开发中 = 50%
- 待开发 = 0%

每次更新开发计划时同步更新完成度。

输出计算过程。

【Git里程碑规则】

达到以下条件之一：

1. 一个 Sprint 完成
2. 一个 P0 功能完成并验收通过
3. 数据结构发生变化
4. 页面结构发生变化

自动建议：

Git Commit

并生成：

- Commit Message
- 本次变更摘要
- 回滚说明
